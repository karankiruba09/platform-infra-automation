If the goal is to answer "how much of my Avi and NSX-T DFW licensing is actually being used?", the fastest approach is to automate both collections and keep the output in machine-readable files. This post shows exactly how to do that with the Ansible content in this repo.

What This Covers

The repo already contains two small automation bundles:

  • Avi license utilization: queries each Avi controller for licenseusage.
  • NSX-T DFW license utilization: queries each NSX manager at GET /api/v1/licenses/licenses-usage and extracts the DFW feature row.

Both playbooks produce the same two report formats:

  • reports/license_usage_report.json
  • reports/license_usage_report.csv

Prerequisites

  • Ansible installed on the host where you will run the playbooks.
  • Reachability from that host to your Avi controllers and NSX managers.
  • Credentials for Avi and NSX-T.
  • ansible-vault available so passwords are not stored in plaintext.

The Avi playbook uses the community.network collection path already referenced in the project. The NSX-T playbook uses Ansible's built-in uri module, so there is no separate NSX SDK requirement for the basic report.

1. Generate the Avi License Report

Work from:

networking/avi-license-utilization

Start by creating your local inventory and encrypted vault file:

cp inventory.ini.example inventory.ini
cp vault.yml.example vault.yml
ansible-vault encrypt vault.yml
ansible-vault edit vault.yml

Set the Avi password in the vault file:

vault_avi_password: your-avi-password

The non-secret username is already kept in vars.yml:

avi_username: admin

Populate inventory.ini with your Avi controllers under the [avi_controllers] group:

[avi_controllers]
avi-controller-01.example.local
avi-controller-02.example.local
avi-controller-03.example.local

Run the collection:

ansible-playbook -i inventory.ini license_usage.yml --ask-vault-pass

The Avi playbook queries the licenseusage endpoint on each controller and writes a per-controller summary with these fields:

  • controller
  • host
  • status
  • endpoint
  • consumed_service_cores
  • licensed_service_cores
  • se_units_consumed
  • error

If a controller cannot be queried, the playbook still records the failure and marks the row with status: failed. That makes the report safer to automate because one bad endpoint does not kill the whole run.

2. Generate the NSX-T DFW License Report

Work from:

networking/nsx-t-dfw-license-utilization

Create the local files the same way:

cp inventory.ini.example inventory.ini
cp vault.yml.example vault.yml
ansible-vault encrypt vault.yml
ansible-vault edit vault.yml

Set the NSX password in the vault:

vault_nsx_password: your-nsx-password

The local variable file already defines the basic connection behavior:

nsx_username: admin
nsx_validate_certs: false

Populate inventory.ini with your managers under [nsx_managers]:

[nsx_managers]
nsx-manager-01.example.local
nsx-manager-02.example.local
nsx-manager-03.example.local

Run the collector:

ansible-playbook -i inventory.ini license_usage.yml --ask-vault-pass

This playbook calls:

GET /api/v1/licenses/licenses-usage

It then selects only the DFW feature entry and writes these fields into the report:

  • manager
  • status
  • endpoint
  • feature
  • vm_usage_count
  • cpu_usage_count
  • ccu_usage_count
  • vcpu_usage_count
  • core_usage_count
  • error

3. Where the Reports Land

After each playbook run, look in the local reports directory. You will get both JSON and CSV versions of the same dataset.

That split is useful:

  • Use JSON if you want to feed the output into other automation, dashboards, or a small parser.
  • Use CSV if you want a quick spreadsheet-style view for operations or license review meetings.

4. How to Combine Both Views

I would not merge the raw files into one mixed schema. Avi and NSX-T DFW are measuring different things, so forcing them into one table adds noise. A better approach is:

  1. Run the Avi collector.
  2. Run the NSX-T DFW collector.
  3. Keep each raw report intact.
  4. Build a small summary layer above them for human review.

For example:

  • Avi gives you service core consumption versus licensed service cores.
  • NSX-T DFW gives you protected footprint by VM, CPU, CCU, vCPU, and CORE usage counts.

That is enough to answer two different planning questions from one reporting cycle: how much load-balancing capacity is being consumed and how large the firewall-protected estate is.

5. What a Real Run Looked Like

Using the latest local report artifacts available when writing this post:

  • The Avi report was generated on March 9, 2026 and returned 9 successful controllers out of 10, with one authentication failure.
  • The NSX-T DFW report was generated on March 10, 2026 and returned 10 successful managers out of 10.

That is exactly why the playbooks record status and error fields instead of assuming every API call will succeed. In practical operations, partial success with clean error reporting is better than a fragile all-or-nothing script.

6. Why This Pattern Works

The useful part is not just the API call. It is the full workflow:

  • Inventory is local and simple.
  • Secrets live in Ansible Vault.
  • Each run creates JSON and CSV outputs.
  • Failures are preserved in the report instead of hidden.
  • The structure is small enough to schedule later with cron, AWX, or another runner.

Closing

If you need a quick, maintainable way to report both Avi license consumption and NSX-T DFW license usage, these two directories are already enough. Start by running each playbook independently, validate the JSON and CSV output, and then build your summary or dashboard on top of those files rather than inside the collection logic.
